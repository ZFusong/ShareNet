# Lessons

- 当快捷键录制需要区分“单次按下”和“长按自动连发”时，不要再用纯时间窗去重代替键盘状态机；应该用 `keydown` / `keyup` 的按下状态配合 `event.repeat`，这样才能既防重复又不误伤重新按下的同一按键。
- 当仓库同时存在 src/preload/index.ts 和遗留的 src/main/preload.js 时，新增桥接 API 不能只改一处，必须确认运行时真正加载的 preload 实现并同步更新。
- 当用户明确收敛“本机配置”和“远端设备配置”的边界时，要立即回头检查数据模型里是否错误引入了 `deviceKey` 一类跨设备维度，避免把本地触发器映射做成全网集中映射。
- 当快捷键录制与对话框复用同一个按键事件时，不能只依赖子组件的 preventDefault；对于 Escape 这类会被对话框框架拦截的键，需要在 Dialog.Content 层显式拦截，避免误关闭弹框。
- 当组合快捷键的展示需要更像设置面板时，应该把整个组合渲染成一个整体 chip，而不是逐个 token 渲染，否则单键组合和重复按键都会在视觉上显得像数据重复。
- 当用户在组合键和单键之间切换录入类型时，不能沿用旧的 key/code/ctrlKey 状态；应当把输入框清空成初始态，再让用户重新录入，避免旧值污染新类型。
- 当 `apply_patch` 连续报 `windows sandbox: setup refresh failed` 时，先把它当成工具侧工作区刷新故障处理，不要反复硬撞同一补丁；改用 PowerShell 做定点替换，比继续重试补丁更稳。
- 当用脚本批量替换 TSX 结构时，不要只看替换结果的外观，必须额外检查是否多留了孤立的大括号或断开的 JSX 结构；先做尾部语法扫描，再跑构建验证。
- 当仓库已经提供现成的 Dialog / AlertDialog / Select 封装时，不要先新增一层业务包装再去替换页面；应直接复用现有封装，避免多出无意义的中间组件。
- 当用户明确说“不需要兼容旧鼠标数据”时，不要再保留迁移、兜底或 legacy 类型分支；应直接收口到新模型，避免旧语义继续污染类型和执行路径。
- 当录制流程需要“结束/确认当前位置”时，不要用任意空白左键当作提交信号；应该提供显式确认按钮，否则用户点空白也会被误判为完成。
- 当录制弹框在录制态需要继续捕获鼠标移动时，必须显式禁用外部点击关闭；否则鼠标点空白区域会把对话框当成普通关闭事件处理。
- 当录制弹框需要“点击屏幕取点”时，空白区域和对话框内容必须分流处理：空白点击负责取点，对话框内部交互必须忽略，否则按钮会被误当成录制事件。
- 当录制态要求左键反复更新坐标时，不要把第一次点击当成最终提交；应该把点击定义成“采样点更新”，由显式结束按钮退出录制态。
- 当录制器需要在对话框内继续采样全局屏幕点击时，不要依赖 document 事件去猜外部点击；直接在弹框上方放一个专用采样层更稳定。
- 当录制器同时需要“当前坐标”和“点击坐标”时，必须拆成两套状态：前者跟随鼠标移动，后者只在左键采样时更新；不要让一个字段同时承担展示和提交职责。
- 当录制器需要“当前坐标”和“点击坐标”同时存在时，开始录制按钮本身就应初始化一份当前坐标；不要等第一帧鼠标移动才给默认值，否则用户会看到空状态。
- 当录制器从某个预设步骤打开时，不要把“开始录制”按钮本身的点击坐标当成采样结果；按钮只负责切换到录制态，初始坐标应来自目标步骤，真正的坐标更新必须等录制态里的左键点击。
