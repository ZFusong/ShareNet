# ShareNet 新操作逻辑（V2）

## 1. 已确认的目标逻辑

- 操作台只负责发送触发器 `triggerKey`。
- 操作台不负责选择场景、软件预设、键盘预设，也不负责判断远端设备内部如何映射。
- 配置中心负责维护：
  - 软件预设
  - 键盘预设
  - 场景编排
  - 本机触发器绑定（`triggerKey -> scene`）
- 发送流程：
  1. 操作台选择目标设备（选择设备/设备分组）
  2. 输入或选择触发器 `triggerKey`
  3. 发送 `EXECUTE_TRIGGER`
  4. 目标机按自己的本地触发器绑定查 scene 并执行
  5. 回执 `EXECUTE_TRIGGER_RESULT`

## 2. 当前收敛后的实现原则

- 触发器绑定不再区分设备。
- 发送触发器时只发送 `triggerKey`。
- 接收端仅根据本机 `triggerKey -> scene` 绑定执行场景。
- 操作台不再基于本地配置中心里的设备级绑定做发送前预检查。
- 操作台里的触发器候选仅作为本机已配置 key 的便捷输入，不代表远端一定支持。

## 3. 仍需继续收敛的点

- 鼠标动作应主要放在“场景步骤”中；键盘预设侧重键盘动作。
- 场景编排 UI 目前还偏“勾选预设”，后续应升级为真正的步骤流编辑器。
- 如果以后需要“远端是否支持该 triggerKey”的精确预检，需要额外做设备能力上报，而不是回到设备级触发器绑定模型。

## 4. 建议后续收敛顺序

1. 把场景编排 UI 升级成真正的步骤编辑器，支持软件、键盘、鼠标、延迟。
2. 统一延迟表达方式，优先使用显式 `delay` 步骤。
3. 如需远端候选 trigger 交集，再增加设备上线时的 trigger 能力同步。
